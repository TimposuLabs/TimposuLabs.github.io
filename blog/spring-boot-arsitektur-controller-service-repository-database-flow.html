Arsitektur Spring Boot - Controller, Service, Repository dan Database

March 15, 2025 ¬∑ 5 min read
Ucup TopekoX
TimposuLabs creator

Spring Boot üçÉ adalah salah satu framework paling populer untuk membangun API RESTful dan Microservices di Java. Spring Boot menyederhanakan pengembangan aplikasi Java dengan menyediakan environtment yang telah dikonfigurasi sebelumnya, menyederhanakan penulisan kode, dan memastikan scalability dan maintainability.

üìå Arsitektur Spring Boot‚Äã

Pada tutorial ini kita akan meng-eksplore Spring Boot Layered Architecture, dan bagaimana data mengalir di antara masing-masing komponen layer tersebut. Sebelumnya, kenapa kita harus menggunakan arsitektur tersebut:

  • ‚òëÔ∏è Struktur kode program lebih terorganisasi.
  • ‚òëÔ∏è Scalability: Membuat aplikasi lebih mudah dikembangkan/diperluas.
  • ‚òëÔ∏è Maintainability: Masing-masing layer memiliki tugasnya masing-masing.
  • ‚òëÔ∏è Faster Development: Menyederhanakan interaksi API dan database.

üîÅ Spring Boot Layered Architecture‚Äã

Arsitektur Spring Boot didasarkan pada pendekatan berlapis/layered, di mana setiap lapisan/layer bertanggung jawab atas bagian tertentu dari aplikasi.

1Ô∏è‚É£ Client Layer (Pengguna API)‚Äã

Client Layer adalah entitas eksternal dari aplikasi Spring Boot (browser, aplikasi seluler, Postman, aplikasi frontend) yang berinteraksi dengan API.

Client akan:

  • ‚úîÔ∏è Mengirim HTTP Requests (GET, POST, PUT, DELETE)
  • ‚úîÔ∏è Menerima API Responses (JSON format)

Contoh client:

  • ‚úîÔ∏è Frontend apps (React, Angular, Vue.js)
  • ‚úîÔ∏è Mobile apps (Android, iOS)
  • ‚úîÔ∏è API testing tools (Postman, cURL)

2Ô∏è‚É£ Controller Layer (Menangani HTTP Requests & Responses)‚Äã

Controller Layer bertindak sebagai pintu masuk untuk permintaan API. Dia bertanggung jawab untuk memproses permintaan HTTP yang masuk/request dan mengembalikan response yang sesuai.

üìå Tugas Controller Layer‚Äã

  • ‚úîÔ∏è Menerima requests dari client (@GetMapping, @PostMapping, dsb.).
  • ‚úîÔ∏è Melakukan validasi dari data yang diinput.
  • ‚úîÔ∏è Memanggil Service Layer untuk business logic.
  • ‚úîÔ∏è Mengembalikan response HTTP yang sesuai.

üìå Contoh Controller‚Äã

@RestController
@RequestMapping("/api/person")
public class PersonController {

    private final PersonService personService;

    public PersonController(PersonService personService) {
        this.personService = personService;
    }

    // ‚úÖ GET: get all person (200 OK)
    @GetMapping
    public ResponseEntity<List<PersonDTO>> getAll() {
        return ResponseEntity.ok(personService.getAll());
    }
}

üöÄ Controller bukan untuk bussines logic. Dia hanya mengatur permintaan ke method pada layer service yang sesuai.

3Ô∏è‚É£ Service Layer (Business Logic Processing)‚Äã

Service Layer bertanggung jawab untuk menerapkan bussines logic dan memproses data sebelum mengirimkannya ke client.

üìå Tugas Service Layer‚Äã

  • ‚úîÔ∏è Mengimplementasikan business rules dan logic.
  • ‚úîÔ∏è Menangani transactions.
  • ‚úîÔ∏è Memanggil Layer Repositori untuk interaksi ke database.
  • ‚úîÔ∏è Menggunakan DTO (Data Transfer Objek) untuk struktur data.

üìå Contoh Service‚Äã

@Service
public class PersonService {

    private final PersonRepository personRepository;
    private final PersonMapper personMapper;

    public PersonService(PersonRepository personRepository, PersonMapper personMapper) {
        this.personRepository = personRepository;
        this.personMapper = personMapper;
    }

    public List<PersonDTO> getAll() {
        return personRepository.findAll().stream()
                .map(personMapper::toPersonDTO)
                .toList();
    }
}

üöÄ Service layer memastikan bahwa Controller tidak mengandung bussines logic. Dia bertanggung jawab untuk menangani business operations pada aplikasi.

4Ô∏è‚É£ Repository Layer (Database Access Layer)‚Äã

Repository Layer bertanggung jawab untuk berkomunikasi dengan database.

Hal-hal yang perlu diperhatikan:

  • ‚òëÔ∏è Menggunakan Spring Data JPA untuk melakukan operasi CRUD.
  • ‚òëÔ∏è Menggunakan @Repository annotation untuk menandai bahwa dia adalah DAO (Data Access Object).
  • ‚òëÔ∏è Mengimplementasikan queri database menggunakan JPA, Hibernate, atau Native SQL.

üìå Contoh Repository‚Äã

public interface PersonRepository extends JpaRepository<Person, Long> {
}

üöÄ Spring Data JPA mengurangi penggunaan kode CRUD boilerplate dengan menyediakan method yang telah dibuat oleh Spring Data JPA secara otomatis seperti findAll(), save(), deleteById() dll.

5Ô∏è‚É£ Model Layer (Entity & DTO Representation)‚Äã

Model Layer mewakili tabel yang ada dalam database dan memastikan enkapsulasi data.

Hal-hal yang diperhatikan:

  • Entity dipetakan ke tabel database.
  • DTO (Objek Transfer Data) hanya membantu mentransfer data yang diperlukan.

üìå Contoh Entity‚Äã

@Entity
public class Person {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String firstName;

    private String lastName;

}

üöÄ Entity tidak boleh diekspose secara langsung dalam response API. Sebagai gantinya, kita menggunakan DTO.

üìå Contoh DTO menggunakan Java Record‚Äã

public record PersonDTO(Long id, String firstName, String lastName) {
}

üöÄ Menggunakan record untuk DTO, akan membuat immutable dan kode lebih clean.

6Ô∏è‚É£ Database Layer‚Äã

Database Layer untuk menyimpan dan mengambil data menggunakan framework persistence Spring Boot.

Hal-hal yang diperhatikan:

  • Menggunakan database relasional (MySQL, MariaDB, PostgreSQL, Oracle, Ms SQL, H2, dll.).
  • Menggunakan JPA dan Hibernate untuk mengelola entity mapping.
  • Menjalankan query menggunakan operasi CRUD.

üìå Contoh Konfigurasi Database pada application.properties‚Äã

spring.datasource.url=jdbc:mariadb://localhost:3306/belajar
spring.datasource.driverClassName=org.mariadb.jdbc.Driver
spring.datasource.username=root
spring.datasource.password=yourpassword
spring.jpa.database-platform=org.hibernate.dialect.MariaDBDialect
spring.jpa.show-sql=true
spring.jpa.hibernate.ddl-auto=update

üöÄ Konfigurasi di atas menggunakan database MariaDB. Untuk database lain tentunya memiliki konfigurasi yang berbeda.

‚úÖ Bagaimana Alur Data Mengalir?‚Äã

  • 1Ô∏è‚É£ Client mengirim request HTTP ke layer Controller.
  • 2Ô∏è‚É£ Layer Controller melakukan validasi request dan meneruskan ke layer Service.
  • 3Ô∏è‚É£ Layer Service memproses bussines logic dan memanggil layer Repository.
  • 4Ô∏è‚É£ Layer Repository mengambil atau memperbaharui data yang ada di Database.
  • 5Ô∏è‚É£ Layer Model memetakan/mapping database record ke Java object.
  • 6Ô∏è‚É£ Data yang sudah diproses dikirim kembali ke layer Service, lalu ke layer Controller, dan akhirnya dikembalikan ke Client sebagai response API.

üéØ Kesimpulan‚Äã

  • ‚úÖ Memisahkan layer arsitektur berdasarkan fungsinya, sehingga penanganan permasalahan antar layer lebih baik.
  • ‚úÖ Menggunakan Controller, Service, Repository, Model dan Database layer.
  • ‚úÖ Memastikan kode program lebih clean, maintainability dan scalability.
  • ‚úÖ Menggunakan Spring Data JPA untuk interaksi ke database.
